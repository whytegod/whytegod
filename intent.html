Intent Model

This document defines the intent model used by the Whytegod protocol. An intent represents a user’s desired outcome, independent of execution steps, infrastructure, or counterparties.

Definition

An intent is a declarative statement of what a user wants to achieve, not how the action should be performed.

Intents are designed to be deterministic, machine-readable, and policy-constrained.

Design Principles

  • Outcome-focused, not procedural
  • Independent of execution routes
  • Explicit constraints and limits
  • Auditable and verifiable

Intent Structure

A valid intent MUST specify:

  • the desired outcome
  • constraints (limits, time, risk)
  • acceptable failure conditions

An intent MUST NOT specify:

  • specific exchanges or venues
  • manual routing logic
  • custodial assumptions

Validation

Intents are validated before execution to ensure:

  • schema correctness
  • internal consistency
  • compliance with protocol rules

Invalid intents MUST be rejected prior to execution planning.

Execution Independence

The same intent may produce different execution plans depending on:

  • available liquidity
  • network conditions
  • policy constraints

This separation ensures flexibility without sacrificing user intent.

Protocol Invariants

  • User intent is the primary authority
  • Execution logic is replaceable
  • No hidden interpretation of intent
  • All constraints are explicit
