Governance

This document defines how the Whytegod protocol evolves over time. Governance prioritizes stability, predictability, and minimal change.

Whytegod is designed to be conservative by default. Changes are expected to be rare, deliberate, and well-justified.

Principles

  • Protocol stability over feature velocity
  • Backward compatibility whenever possible
  • Explicit, documented changes
  • No emergency or unilateral upgrades

Change Proposals

All protocol changes MUST be proposed in writing. Proposals SHOULD include:

  • clear motivation
  • technical specification
  • impact analysis
  • security considerations

Proposals are reviewed publicly and discussed openly.

Acceptance Criteria

A proposal MAY be accepted if it:

  • preserves protocol invariants
  • does not introduce hidden trust assumptions
  • improves clarity, safety, or interoperability

Acceptance does not imply immediate implementation.

Versioning

Protocol versions are explicit and immutable once released. Implementations may choose which versions to support.

Deprecated behavior SHOULD remain documented.

Non-Goals

Governance does NOT include:

  • token voting
  • profit distribution
  • operational control of implementations

Final Authority

The protocol specification itself is the final authority. No individual, organization, or implementation has special control.
